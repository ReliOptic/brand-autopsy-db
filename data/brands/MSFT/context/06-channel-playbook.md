# 06. 채널 플레이북 — Microsoft (MSFT)

## 채널별 전략

### 채널 1 — LinkedIn

**역할:** B2B 리드 생성, 임원·IT 의사결정자 도달, 사고 리더십(Thought Leadership) 구축

**핵심 오디언스:** CIO·CTO·CISO, IT 관리자, 개발자 (시니어 이상), HR·생산성 담당자

**콘텐츠 믹스:**
| 유형 | 비중 | 예시 |
|------|------|------|
| 고객 사례 연구 | 30% | "XX기업이 Azure로 TCO 30% 절감한 방법" |
| 제품 업데이트·기능 소개 | 25% | "Copilot Wave 2: Teams에서 달라지는 것" |
| 산업 인사이트·리포트 | 20% | "Work Trend Index 2024 주요 발견" |
| 임원 퍼스널 브랜딩 | 15% | Satya Nadella 또는 한국 지사장 게시물 |
| 채용·문화 | 10% | Microsoft 한국 직원 스토리 |

**최적 포스팅 시간:** 화·수·목 오전 7~9시 (KST 기준, 출근 전 스크롤)

**형식 가이드:**
- 텍스트 포스트: 첫 3줄이 훅. 줄바꿈 자주. 이모지 1~2개 허용.
- 카루셀(슬라이드): 표지 강렬한 수치 → 내용 → CTA. 10슬라이드 이하.
- 영상: 30초~3분. 자막 필수(무음 자동재생). 첫 5초에 핵심 메시지.
- 문서(PDF): 백서·케이스 스터디 공유 시. 리드 수집 랜딩페이지 연결.

**KPI:** CPL(리드당 비용), 임원 프로필 조회수, 기업 페이지 팔로워 증가율, 웨비나 등록 전환율

### 채널 2 — YouTube

**역할:** 제품 데모, 교육·튜토리얼, 브랜드 스토리텔링, SEO 기반 인바운드 트래픽

**핵심 오디언스:** 개발자, IT 관리자, 제품 평가 중인 의사결정자, Microsoft Learn 학습자

**콘텐츠 믹스:**
| 유형 | 비중 | 예시 |
|------|------|------|
| 제품 데모·하우투 | 35% | "Azure OpenAI Service 5분 시작 가이드" |
| 이벤트·컨퍼런스 키노트 | 25% | Microsoft Ignite, Build 세션 풀 영상 |
| 고객 스토리 영상 | 20% | 기업 AI 전환 다큐멘터리 형식 |
| 브랜드 캠페인 영상 | 15% | "Be ready for what's next" 시리즈 |
| Microsoft Mechanics (기술 딥다이브) | 5% | 엔지니어 대상 아키텍처 설명 |

**썸네일 규칙:**
- 1280×720px, 텍스트 최소화 (제목 3~5단어)
- 브랜드 블루(`#0067B8`) 또는 다크 네이비(`#243A5E`) 배경
- 인물 사진 포함 시 CTR 상승 경향 (추정)

**SEO 전략:**
- 제목: "[제품명] + [구체적 작업] + [시간/결과]" 형식 (예: "Copilot으로 Excel 분석 5분 만에")
- 설명란 첫 2줄에 핵심 키워드 + 타임스탬프 필수
- 관련 Microsoft Learn 링크, 무료 체험 CTA 링크 항상 포함

**KPI:** 조회수, 평균 시청 시간, CTR, Microsoft Learn 유입 비율

### 채널 3 — Microsoft Tech Blog / Microsoft Learn

**역할:** 개발자·IT 전문가 신뢰 구축, SEO 기반 오가닉 트래픽, 제품 교육 허브

**핵심 오디언스:** 개발자, 클라우드 아키텍트, IT 관리자, Microsoft 자격증 학습자

**콘텐츠 전략:**
- **Microsoft Tech Community Blog**: 제품팀 엔지니어가 직접 작성. 기능 출시 배경, 기술 결정 이유, 로드맵 힌트 포함. 마케팅 언어 최소화. 코드 예시 필수.
- **Microsoft Learn**: 모듈형 학습 경로. 완료 시 배지·자격증 연결. 무료 접근. 검색 트래픽 기반 인바운드 핵심 채널.
- **Azure Architecture Center**: 레퍼런스 아키텍처 다이어그램 제공. 의사결정자가 기술 검토 시 참조하는 문서.

**글쓰기 규칙:**
- 코드 블록은 복사 버튼 포함
- 전제 조건(Prerequisites) 섹션 항상 첫머리에
- "시간 소요 예상: X분" 표기 (Microsoft Learn 표준)
- 관련 문서·다음 단계 링크 항상 말미에

**KPI:** 페이지뷰, 학습 모듈 완료율, 자격증 응시 전환율, 오가닉 검색 순위

**핵심 오디언스:** 개발자, 테크 저널리스트, 오픈소스 커뮤니티, 얼리어답터

**콘텐츠 믹스:**
| 유형 | 비중 | 예시 |
|------|------|------|
| 제품 출시 속보 | 30% | "Copilot in [제품] 오늘 GA!" |
| GitHub 오픈소스 활동 | 25% | "VSCode 1.90 릴리즈 노트" |

| 이벤트 라이브 커버리지 | 10% | Build·Ignite 실시간 트윗 |

**톤 규칙 (X 한정):**

**금지:**
- 정치적 발언
- 경쟁사 직접 비하
- 공식 발표 전 제품 기능 언급

**KPI:** 임프레션, 인게이지먼트율, 개발자 계정 팔로워 증가, 커뮤니티 언급 감성

### 채널 5 — 이메일 (Email Marketing)

**역할:** 기존 고객 리텐션, 업셀·크로스셀, 제품 업데이트 전달, 이벤트 드라이브

**핵심 오디언스:** 기존 Microsoft 365·Azure 구독자, 무료 체험 사용자, 이벤트 참석자

**이메일 유형별 전략:**

| 유형 | 발송 주기 | 핵심 목적 | 성과 지표 |
|------|-----------|-----------|-----------|
| 제품 업데이트 뉴스레터 | 월 1회 | 기능 인지·활성화 | 오픈율, 클릭률 |
| 온보딩 시퀀스 | 가입 후 D+1, D+3, D+7, D+14 | 활성화율 향상 | 첫 핵심 기능 사용률 |
| 이벤트 초대 | 이벤트 2주 전, 1주 전, 당일 | 등록 전환 | 등록률, 참석률 |
| 갱신 리마인더 | 만료 30일, 14일, 7일, 1일 전 | 이탈 방지 | 갱신율 |
| 업셀 캠페인 | 분기별 | ARPU 향상 | 전환율, 업셀 수익 |

**이메일 카피 원칙:**
- 제목줄: 40자 이하. 수치·혜택·긴급성 중 하나 포함.
- 프리헤더: 제목 보완. 40~90자.
- 본문 첫 문장: 핵심 혜택 직접 전달. "안녕하세요" 인사 금지.
- CTA 버튼: 1개 주요 CTA. 텍스트는 동사+명사 (예: "무료로 시작하기", "데모 예약하기")
- 이미지 크기: 600px 너비, 이미지 없이도 의미 전달되도록 alt 텍스트 필수

**핵심 오디언스:** 파트너사 기술 임원, 개발자, MVP 커뮤니티, 미디어

## 채널 시너지 구조

```
[이벤트 발표 (Ignite/Build)]
        │
        ▼
[YouTube 키노트 클립] → [X 실시간 커버리지] → [LinkedIn 요약 카루셀]
        │
        ▼
[Tech Blog 기술 상세 포스트] → [Microsoft Learn 모듈 업데이트]
        │
        ▼
[이메일 뉴스레터 "이번 달 주요 업데이트"]
        │
        ▼
[LinkedIn 고객 사례 연구] (발표 후 8~12주 후)
```

**시너지 원칙:**
1. 모든 대형 발표는 이벤트 → 영상 → 블로그 → 이메일 → SNS 순서로 확산
2. 하나의 콘텐츠를 채널별 포맷으로 재가공 (1 Source → 5 Formats)
3. Tech Blog가 SEO 앵커. 다른 채널에서 Blog로 트래픽 유입.
4. LinkedIn이 B2B 리드 수집 최종 착지점. 모든 채널 CTA를 LinkedIn 이벤트 또는 랜딩페이지로 연결.

| P1 (긴급 대응) | 제품 보안 취약점 공개, 주요 파트너 계약 분쟁 | 4시간 이내 |

### P0 위기 대응 메시지 프레임워크

**단계 1 — 인지 (1시간 이내):**
> "현재 [문제]가 발생했음을 인지하고 있습니다. 원인 파악 중이며, 업데이트는 [채널]을 통해 [주기]마다 공유하겠습니다."

**단계 2 — 조치 공유 (발생 후 4시간):**
> "현재까지 확인된 사실: [사실]. 조치 중인 내용: [행동]. 영향받는 범위: [범위]. 예상 복구 시간: [시간]."

**단계 3 — 사후 보고 (복구 후 24~48시간):**
> "서비스가 복구되었습니다. 원인: [근본 원인]. 재발 방지 조치: [구체적 변경사항]. 보상 정책: [SLA 크레딧 등]."

**위기 대응 채널 우선순위:**
1. Azure Status Page (status.azure.com) — 기술 상태 실시간 업데이트 (공식)
2. X (@AzureSupport, @MicrosoftHelps) — 공개 소통
3. 고객 직통 이메일 — EA 고객 우선 통보
4. LinkedIn — 임원 메시지, 사후 투명성 보고
5. 보도자료 — P0 수준에서만

**위기 시 절대 금지:**
- "사소한 문제입니다" 축소
- 수동태로 책임 회피 ("문제가 발생되었습니다")
- 원인 파악 전 확정 발언
- 경쟁사 비교로 주의 분산
- SNS 댓글 삭제 또는 비공개 전환

### Steal Sheet — 채널 플레이북

1. **"1 Source → 5 Formats" 콘텐츠 재활용 엔진"** — Microsoft는 Ignite 키노트 하나를 YouTube 풀 영상 + 클립 시리즈 + Tech Blog 요약 + LinkedIn 카루셀 + 이메일 뉴스레터로 5배 확장한다. 콘텐츠 생산량이 아니라 재활용 설계가 효율의 핵심. 한국 마케팅팀에서 훔칠 시스템: 콘텐츠 제작 전 "이 콘텐츠를 몇 개 채널로 재가공할 수 있는가?"를 먼저 묻는 포맷 계획 시트를 도입.

2. **"위기 대응 3단계 스크립트를 미리 만들어두기"** — Microsoft는 Azure 장애 발생 시 1시간 이내 공개 커뮤니케이션을 할 수 있는 이유가 위기 등급별 메시지 프레임워크가 사전에 준비되어 있기 때문이다(추정). 한국 브랜드에서 훔칠 원칙: 위기가 오기 전에 "우리 브랜드에서 발생 가능한 P0 시나리오 3가지"와 "각각의 첫 번째 공개 메시지 초안"을 지금 만들어두어라. 위기 속에서 메시지를 만들면 반드시 실수한다.

3. **"Status Page = 신뢰 자산"** — Microsoft는 status.azure.com이라는 공개 상태 페이지를 운영해 서비스 가용성을 실시간으로 공개한다. 숨기면 소문이 더 크게 번진다. 공개 투명성이 장기적 신뢰를 만든다. 한국 SaaS에서 훔칠 원칙: 내 서비스의 공개 Status Page를 만들어라 (Statuspage.io, BetterUptime 등 무료/저가 도구로 30분 안에 가능). 평소에는 "100% 운영 중" 표시만으로도 전문성을 전달하고, 장애 시 투명한 소통 채널로 활용한다.
